## 4. Data Consistency & Quality Checks
We evaluated the data against four critical logical rules:
1. **Mathematical Check (`Sales - Cost = Gross Profit`):** 
   * *Status:* **Pass.** 0 violations detected.
2. **Negative Values Check (Sales, Units, Cost, Profit < 0):** 
   * *Status:* **Pass.** 0 negative values found.
3. **Temporal Inversion Check (`Ship Date < Order Date`):** 
   * *Status:* **Pass.** 0 occurrences where a shipment preceded the order.
4. **Logistical Feasibility Check (Shipping Lead Times):**
   * *Status:* **Failure / Core Data Anomaly.** 
   * Order dates span **2024-01-02** to **2025-12-31**.
   * Ship dates span **2026-06-30** to **2030-06-28**.
   * **Result:** The minimum shipping lead time is **904 days (2.5 years)**, with a median of **1,274 days (3.5 years)**. 

---

## 5. Potential Data Quality Issues & Risks
* **The "3-Year Shipping" Anomaly:** In a real supply chain, chocolate would spoil, cash would be locked up, and customers would cancel. This is likely an IT integration logging error (e.g., mismatched system years or synthetic date offsets).
* **Extreme Division Imbalance:** Because Chocolate represents **96.57%** of the rows, any predictive model trained on this data will be biased toward chocolate distribution paths. Sugar represents only 40 records (0.39%), making it statistically vulnerable to noise.
* **Low Product Catalog Size vs. High Order Volume:** 15 SKUs map to 10,194 records, which indicates high density and repeated routes. This simplifies clustering but risks overfitting.

## 6. Recommended Preprocessing Pipeline
To prepare this dataset for predictive modeling and scenario simulation, the following steps are required:
1. **Lead Time Normalization:** Subtract `Order Date` from `Ship Date` to calculate `lead_time_days`. Shift or scale the absolute years (e.g., reducing the years offset by 3) to represent a standard shipping time window (2-7 days) for the UI simulator to look realistic, or document the scale factor.
2. **Geocoding Enrichment:** Create a coordinate mapping for customer states/cities center points to calculate physical distance to the 5 factories.
3. **One-Hot Encoding:** Encode low-cardinality categorical fields: `Ship Mode`, `Region`, and `Division`.
4. **Stratified Splitting:** Due to the extreme imbalance in the `Division` column, use a stratified train-test split to ensure that `Sugar` and `Other` products are evenly represented in both datasets.
5. **Feature Scaling:** Standardize the numerical features (`Units`, `Sales`, `Cost`, `Distance`) using `StandardScaler` to ensure distance metrics do not dominate regressor models.
